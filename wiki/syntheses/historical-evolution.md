# Historical Evolution Matrix

The Systems EM framework evolved through several recognizably different stages. This page is a lineage map showing how earlier execution-centric concepts widened into a broader sociotechnical model.

> [!NOTE]
> Legacy files are preserved in `wiki/legacy/`. They remain useful for historical comparison, but they should not be treated as the best current operating reference.

## V1 to V2: From Execution Model to Multi-Block Framework

The original V1 model focused almost entirely on execution conditions: clarity, focus, skill, coordination, quality, and feedback speed. Its main limitation was that it could describe team mechanics without adequately representing the strategic, human, and structural forces that often dominate those mechanics.

V2 widened the scope by embedding execution inside a larger set of blocks.

| V1 Condition (Legacy) | V2 Relocation |
|-----------------------|---------------|
| [Clarity (OM1)](../legacy/clarity.md) | Nested into Execution and distinguished from strategy/value questions |
| [Focus (OM2)](../legacy/focus.md) | Nested into Execution and later coupled more explicitly with batch size |
| [Skill (OM3)](../legacy/skill.md) | Nested into Execution and read alongside team composition |
| [Coordination (OM4)](../legacy/coordination.md) | Nested into Execution but increasingly bounded by structure |
| [Quality (OM5)](../legacy/quality.md) | Nested into Execution |
| [Feedback Speed (OM6)](../concepts/feedback-speed.md) | Shifted toward Learning and Adaptation |

## V2 to the Expanded Current Formulation

V2 established the five-block structure (`V`, `P`, `E`, `A`, `L`) but still expressed many ideas in compact or assertive shorthand. The later formulation made several moves:

- more explicit parameterization of the blocks;
- stronger emphasis on queueing and variability;
- clearer treatment of people health and measurement reliability;
- more explicit diagnostic order and historical comparison paths.

## What Changed in Spirit

The biggest change is not just more blocks or more math. It is a change in what the framework is trying to do.

Earlier versions often read like a compact control theory for execution. The newer material reads more like a sociotechnical operating model:
- less confidence that one score explains everything;
- more attention to context, safety, and structural limits;
- more explicit treatment of heavy tails, adaptation, and blocked local optimization.

That makes the current wiki more useful for real diagnosis, while the legacy pages remain valuable for understanding how the model got there.
